# 会员活动管理系统说明文档

> 本文档基于当前 `feature/proxy_jane_20260302` 分支的实际代码，系统化梳理"会员活动（积分抽奖）"模块的架构、数据模型、接口契约与前端调用方式，供前后端联调、二次开发与测试参考。

## 一、系统概览

会员活动模块依托统一 App 框架（认证、路由、HTTP、状态管理、App Proxy 等），在 Shopify 商家后台实现"积分抽奖"活动管理能力：

- 活动 CRUD：列表 / 新建 / 详情 / 更新 / 删除 / 复制
- 奖品配置：无中奖 / 产品折扣 / 订单折扣 / 礼品卡 / 积分（共 5 类）
- 数据看板：按时间区间统计参与人数、各类奖品中奖数与环比
- 中奖记录：多维筛选、按邮箱搜索、CSV 导出
- 前台店铺侧通过 App Proxy 与业务后端 / Smile / Shopify Customer 通信

模块涉及代码主要集中在以下位置：

| 类别                           | 路径                                                                |

| 业务 API                       | `app/services/activity.ts`                                          |
| 业务类型                       | `app/types/activity.ts`                                             |
| HTTP 客户端                    | `app/utils/http.ts`                                                 |

## 三、运行架构与请求链路

会员活动有 **三条独立请求链路**，需要分别理解。

### 1）后台管理端（Embedded Admin）

`app/pages/app/activities/*` 直接通过 `http` 客户端调用业务后端，请求自动携带 Shopify Session Token（`shopify.idToken()`），便于后端基于 JWT 校验店铺身份。

```
Admin 页面 → activityApi → http (Bearer <idToken>) → APP_CONFIG.api.rewardsBackendAPI
```

`APP_CONFIG.api.rewardsBackendAPI` 当前值（见 `app/config/index.ts`）：

```
https://shopify-activity.ecolifeglobal.cn
```

### 2）前台店铺端（Theme / Storefront）

主题侧请求通过 Shopify App Proxy 进入 App，再由 App 转发到外部服务：

```
Theme JS → /apps/<proxy_prefix>/rewards/api      → 业务后端 (Backend API)
                                       /smile    → Smile API
                                       /customer → Shopify Admin GraphQL (customer mutations)
```

App Proxy 路由由 `app/utils/proxy.server.ts` 的 `createProxyAction` 工厂统一生成，自带：

- Shopify App Proxy 签名验签（开发环境跳过，并打 warn 日志）
- endpoint 白名单（支持 `/path/*` 通配）
- method 限制（仅 `GET` / `POST`）
- 超时与错误统一化（`TimeoutError` → 504，`HttpError` → status，其余 → 500）
- 开发环境自动附加 CORS 头
- 统一响应结构 `{ code, message, data }`

### 3）Admin 内部 API（`/app/pages/api/*`）

供管理端组件直接调用 Shopify Admin GraphQL，不走业务后端：

- `POST /api/upload-image`：浏览器表单上传 → Shopify Files（staged upload → `fileCreate` → 轮询永久 URL）
- `POST /api/resource-details`：根据 GID 回显已选 Product / ProductVariant / Collection

## 五、HTTP 客户端约定

> 文件：`app/utils/http.ts`

### 5.1 统一响应结构

```ts
interface ApiResponse<T> {
  code: number // 200 = 业务成功；其余抛 BusinessError
  message: string
  data: T
}
```

`http` 在 `code !== 200` 时抛出 `BusinessError(code, message)`，调用方拿到的是已经解包的 `data`。

### 5.2 自动行为

- 相对路径自动拼上 `APP_CONFIG.api.rewardsBackendAPI`。
- 自动注入 `Content-Type: application/json` 与 `Authorization: Bearer <shopify.idToken()>`（如可获取）。
- 默认超时 `APP_CONFIG.api.timeout`（30s），可通过 `config.timeout` 覆盖。
- 非 2xx HTTP 状态 → `HttpError`；超时 → `TimeoutError`；`code !== 200` → `BusinessError`。
- 三类错误默认会通过 `shopify.toast.show(..., { isError: true })` 弹出（可用 `skipErrorToast: true` 跳过）。
- 支持 `signal: AbortSignal` 取消。

### 5.3 API

```ts
http.get<T>({ url, params?, headers?, signal?, timeout?, skipErrorToast? })
http.post<T>({ url, data?, ... })
http.put<T>({ url, data?, ... })
http.delete<T>({ url, ... })
http.downloadFile({ url, params? })   // 用于 CSV / 文件下载，自动触发浏览器下载
```

### 5.4 拦截器

```ts
import { onRequest, onResponse } from '~/utils/http'

const dispose = onRequest((url, init) => {
  // 修改 url / headers / body
  return [url, init]
})

onResponse((res) => {
  // 可统一改写 res.code / res.data
  return res
})

dispose() // 卸载拦截器
```

## 六、前端如何调用接口

下面是会员活动模块在 Admin 页面中调用的全部业务 API（均封装在 `app/services/activity.ts` 的 `activityApi` 中）。

> 所有方法都返回**已解包**的业务数据；失败会通过抛错的方式提示。

### 6.1 接口清单

| 方法                          | HTTP   | URL                                | 返回                       |
| ----------------------------- | ------ | ---------------------------------- | -------------------------- |
| `activityApi.getList`         | GET    | `/admin/activities`                | `ActivityListResponse`     |
| `activityApi.getById`         | GET    | `/admin/activities/:id`            | `ActivityDetail`           |
| `activityApi.create`          | POST   | `/admin/activities`                | `ActivityListItem`         |
| `activityApi.update`          | PUT    | `/admin/activities/:id`            | `ActivityListItem`         |
| `activityApi.remove`          | DELETE | `/admin/activities/:id`            | `null`                     |
| `activityApi.getDashboard`    | GET    | `/admin/activities/dashboard`      | `DashboardData`            |
| `activityApi.getWinRecords`   | GET    | `/admin/activities/records`        | `WinRecordsResponse`       |
| `activityApi.exportRecords`   | GET    | `/admin/activities/records/export` | `WinRecordsExportResponse` |
| `activityApi.downloadRecords` | GET    | _传入的 url_                       | `void`（触发下载）         |

完整 URL 形如：`https://shopify-activity.ecolifeglobal.cn/admin/activities`。

### 6.2 调用示例

#### a) 列表

```ts
import { activityApi } from '~/services/activity'

const res = await activityApi.getList({ page: 1, limit: 20 })
// res: { total, page, limit, list: ActivityListItem[] }
```

可选参数：`page?: number`、`limit?: number`。`limit` 不传时由后端决定默认值。

#### b) 详情

```ts
const detail = await activityApi.getById('act_123')
```

#### c) 创建 / 更新

```ts
import { validateCompleteActivity } from '~/utils/validation'

const formData: ActivityFormData = {
  /* ... */
}

const error = validateCompleteActivity(formData)
if (error) {
  shopify.toast.show(error, { isError: true })
  return
}

await activityApi.create(formData)
// 或：
await activityApi.update('act_123', formData)
```

**业务约束**：

- `prizes.length` 必须为 **8**（见 `validateCompleteActivity`、`PrizeList` 的 UI 提示与详情页 `canSave`）。
- `prizes` 中各项 `winning_rate` 之和必须 = 100。
- `draw_limit=true` 时 `draw_frequency_day` / `draw_frequency_count` 必填。
- 各类奖品的 `config` 必须符合对应形状（详见第 4.3 节与 `validatePrizeForm`）。

#### d) 删除

```ts
await activityApi.remove('act_123')
```

列表页删除后：若当前页只剩一条且非首页，自动回退一页；否则就地刷新（见 `app/pages/app/activities/index.tsx`）。

#### e) 数据看板

```ts
const data = await activityApi.getDashboard({
  id: 'act_123',
  start_date: '2026-05-01', // 'YYYY-MM-DD'
  end_date: '2026-05-12'
})
```

- 时间维度由前端按"今天 / 过去 7 天 / 过去 30 天 / 自定义"切换，时区使用 `useAppStore` 中的 IANA 时区。
- `*_rate` 为环比百分比（与上一相同长度区间对比），后端可返回 `null` 表示上期无数据。

#### f) 中奖记录

```ts
const data = await activityApi.getWinRecords({
  activity_id: 'act_123',
  prize_type: 'PRODUCT_DISCOUNT_CODE', // 可选；'ALL' 表示不过滤
  status: 'SUCCESS', // 可选；'ALL' 表示不过滤
  search: 'user@example.com', // 可选；按邮箱搜索
  draw_time_start: '2026-05-01',
  draw_time_end: '2026-05-12',
  page: 1,
  limit: 20
})
```

切换筛选条件时（`activeTab` / `status` / `search` / `drawTimeStart` / `drawTimeEnd`），`WinRecords` 组件会自动把 `page` 重置为 1。

#### g) 导出 CSV

```ts
const { download_url, row_count } = await activityApi.exportRecords({
  activity_id: 'act_123',
  prize_type: activeTab,
  status,
  search: search || undefined,
  draw_time_start: drawTimeStart || undefined,
  draw_time_end: drawTimeEnd || undefined

// 业务后端返回的下载直链

// 浏览器自动触发 <a download>，filename 优先取 Content-Disposition
```

### 6.3 在组件中通用模式（useLoading）

`useLoading` 是统一的 loading / error 包装，绝大多数页面都用同一模式：

```ts
import { useLoading } from '~/hooks/useLoading'

const { loading, error, run } = useLoading()
const [data, setData] = useState<ActivityListResponse | null>(null)

const fetchList = useCallback(async () => {
  const res = await run(() => activityApi.getList({ page }))
  if (res) setData(res)
}, [run, page])

useEffect(() => {
  fetchList()
}, [fetchList])
```

如果请求失败，`error` 可直接喂给 `<RetryErrorBanner error={error} onRetry={fetchList} />`。

### 6.4 错误处理

```ts
import { BusinessError, HttpError, TimeoutError } from '~/utils/http'

try {
  await activityApi.create(formData)
} catch (e) {
  if (e instanceof BusinessError) {
    // e.code, e.message ← 业务后端返回的 code/message
  } else if (e instanceof HttpError) {
    // e.status ← 4xx / 5xx
  } else if (e instanceof TimeoutError) {
    // 30s 超时
  }
}
```

默认情况下 `http` 已经弹出 toast；若想自己处理，调用时传 `{ skipErrorToast: true }`。

### 6.5 资源选择器与图片上传（Admin 内部 API）

这部分不走业务后端，直接打到 App 自带的 `/api/*` 路由（仅 Embedded App 后台可用）。

#### `POST /api/upload-image`

`multipart/form-data`，字段 `file`。前端封装：

```ts
import { uploadImageToShopify } from '~/utils/upload'

const url = await uploadImageToShopify(file) // Shopify Files CDN URL
```

服务端流程：`stagedUploadsCreate` → 上传到 Shopify 临时存储 → `fileCreate` → 轮询直到 `fileStatus=READY` 取永久 `image.url`（最多 10 次，每次 1s）。

`ImageUploadField` 组件已封装好 dropzone + 校验（大小默认 5MB、必须 `image/*`）+ 预览 + 删除。

#### `POST /api/resource-details`

入参：

```ts
{ type: 'PRODUCT' | 'COLLECTION', ids: string[] }  // Shopify GID
```

出参：

```ts
{
  resources: Array<
    | { type: 'PRODUCT'; id; title; image?; totalVariants; selectedVariantIds; selectedVariantsCount }
    | { type: 'COLLECTION'; id; title; image? }
  >
}
```

`useResourcePicker` 已经封装好"打开 Shopify ResourcePicker → 回写 ids → 用 `/api/resource-details` 反向回显已选项"的全部流程，业务组件只需关心 `appliesToType` / `appliesToIds` 两个值。

## 七、前台店铺侧（App Proxy）

> 主题端通过 `/apps/<your-proxy-prefix>/rewards/<route>` 命中下列 App Proxy 路由。

所有 Proxy 路由都遵循同一约定：

- **必须 POST**；GET 仅作"占位 loader"用，返回 405 并提示使用 POST。
- 请求体协议：

  ```ts
  interface ProxyRequestBody {
    endpoint: string // 必填，必须以 '/' 开头，并命中白名单
    method?: 'GET' | 'POST' // 默认 GET（仅 GET / POST）
    payload?: unknown // POST 时携带 body
    query?: Record<string, string | number | boolean | null | undefined>
    pathParams?: Record<string, string | number> // 会按值顺序拼到 endpoint 后面
  }
  ```

- 响应统一为 `{ code, message, data }`。
- 开发环境（`NODE_ENV=development`）跳过签名验签并附 CORS 头，便于本地调试。

### 7.1 `POST /proxy/rewards/api` —— 业务后端转发

文件：`app/pages/proxy/rewards/api.tsx`

白名单：

```
/activities/*
/customer
/draws
/customer/draw_records
/check_in
```

附加行为：`forwardProxyParams: true` —— Shopify 注入的 proxy 参数（`shop`、`logged_in_customer_id` 等）会以 `X-Shopify-Proxy` 头透传给业务后端。

主题端示例：

```js
await fetch('/apps/rewards/api', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    endpoint: '/activities/123/state',
    method: 'GET'
  })
})
// 拼接后实际请求：GET https://<backend>/activities/123/state
```

### 7.2 `POST /proxy/rewards/smile` —— Smile API 转发

文件：`app/pages/proxy/rewards/smile.tsx`

白名单：

```
/vip_tiers
/customers
/points_products
/points_transactions
/earning_rules
/reward_fulfillments
```

`Authorization` 头由 App 根据请求中的 `shop` 自动从 `app/config/index.ts` 的 `SMILE_SHOP_TOKENS` 取对应 Bearer Token —— 主题端不需要、也不应该携带任何 Smile 密钥。

```js
await fetch('/apps/rewards/smile', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    endpoint: '/customers',
    method: 'GET',
    query: { external_id: '<shopify_customer_id>' }
  })
})
```

新接入店铺时在 `SMILE_SHOP_TOKENS` 增加 `'xxx.myshopify.com': 'api_...'` 即可。

### 7.3 `GET / POST /proxy/rewards/customer` —— Shopify Customer

文件：`app/pages/proxy/rewards/customer.tsx`

与前两条不同：它**直接使用 Shopify Admin GraphQL**（`authenticate.public.appProxy(request).admin`），没有白名单字段，请求体即是要更新的字段。

- **必须经过 Shopify Customer 登录**（即由 Shopify 自动注入 `logged_in_customer_id` query 参数），否则返回 401。
- `GET`：返回该客户基础资料 + email/sms 订阅状态。
- `POST`：根据请求体里出现的字段同时调用 `customerUpdate` / `customerEmailMarketingConsentUpdate` / `customerSmsMarketingConsentUpdate`。

```ts
// 请求体（任意子集）
interface CustomerUpdateRequest {
  firstName?: string
  lastName?: string
  email?: string
  emailSubscribed?: boolean // SUBSCRIBED / UNSUBSCRIBED（marketingOptInLevel = SINGLE_OPT_IN）
  phone?: string
  smsSubscribed?: boolean
}
```

主题端示例：

```js
const res = await fetch('/apps/rewards/customer', { method: 'GET' })
// → { code:200, data: { firstName, lastName, email, phone, emailSubscribed, smsSubscribed } }

await fetch('/apps/rewards/customer', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ firstName: 'Jack', emailSubscribed: true })
})
```

## 八、业务规则与边界

下列规则在 `app/utils/validation.ts` 与组件层硬性约束，前/后端联调时需保持一致：

### 8.1 基础设置（`baseValidateActivityForm`）

- `name`、`start_time`、`end_time`、`daily_draw_limit`、`consumption_points`、`rules_description` 均必填。
- `start_time < end_time`。
- `rules_description` 是富文本，按"去 HTML 标签 + 去 `&nbsp;` 后非空"判断。
- `draw_limit=true` 时必须给到 `draw_frequency_count`（前端默认 `draw_frequency_day='DAY', draw_frequency_count=1`）。

### 8.2 奖品（`validatePrizeForm`）

- 公共字段：`prize_name`、`winning_rate(0-100)`、`winning_pop_up_message`、`winning_pop_up_cta_text` 必填。
- `prize_type !== 'NO_PRIZE'`：`inventory` 必填且 `inventory >= won_count`。
- `draw_limit=true`：`draw_frequency_count` 必填（默认 7 天 `DAY`）。
- `GIFT_CARD`：`gift_card_value >= 0`。
- `PRODUCT_DISCOUNT_CODE`：`discount.value >= 0`；`applies_to.ids.length >= 1`；`requirement.type !== NONE` 时 `requirement.value` 必填。
- `ORDER_DISCOUNT_CODE`：`discount.value >= 0`；`requirement` 同上。
- `POINTS`：`point_value >= 0`，`point_description` 必填。

### 8.3 整体（`validateCompleteActivity`）

1. 基础设置必须通过；
2. `prizes.length === 8`（业务上"幸运大转盘八宫格"）；
3. 所有奖品 `winning_rate` 之和必须 = 100；
4. 任一不满足都会通过 `shopify.toast.show(..., { isError: true })` 提示。

### 8.4 详情页特殊处理（`PROGRESS` 状态）

- 详情页编辑时 `lockPrizeType` 与 `isProgress` 都设为 `true`：
  - 奖品类型 `<s-select>` 被禁用；
  - "已删奖品列"区域回显 `activity.prizes_deleted`，仅可查看。
- 仍然要求 8 个奖品（详情页 `canSave` 用 `formData.prizes.length === 8` 判断）。

## 十一、扩展指北

- **新增业务接口**：在 `app/services/activity.ts` 加方法即可，URL 使用相对路径；如需自定义 query 序列化，直接使用 `toQueryParams` 帮助函数过滤 `undefined / null`。
- **新增字段**：先改 `app/types/activity.ts`，校验规则放到 `app/utils/validation.ts`，表单 UI 放到 `BasicSettingsForm` / `PrizeForm` 或下游具体 `*ConfigForm`。
- **新增 Proxy endpoint**：在对应 `app/pages/proxy/rewards/*.tsx` 的 `allowedEndpoints` 增加路径（支持 `/foo/*` 通配）；若需要新的外部 API，直接 `createExternalApiClient(...)` 复用工厂能力。
- **新增 Admin GraphQL 接口**：放在 `app/pages/api/*.tsx`，沿用 `authenticate.admin(request).admin.graphql(...)`。
- **多店铺/多环境**：所有外部 baseUrl 与密钥集中在 `app/config/index.ts`，不要散落到组件里。
